# v0.2 Design Draft — `[[wiki-link]]` Cross-Reference Traversal

**Status:** draft (written during v0.1.3 post-merge planning)
**Target release:** v0.2
**Prerequisites:** v0.1.3 (h1-flatten + vault-relative section id) on `main`
**Open questions:** 3 (see §6)

---

## 1. Goal

Let the multi-hop retrieval loop **follow outgoing `[[wiki-links]]`** when
the currently-loaded note references another note that the user's
question is indirectly about. Today the loop can only descend the
folder → note → section tree from the root; after v0.2 it can also
jump sideways along the link graph.

Concrete scenario:

> User asks: "What benchmark did the paper use?"
> Retrieval lands on `Research/paper.md` which contains
> `우리는 [[MMLU]] 벤치마크로 평가했다.` but does not itself describe
> what MMLU is. Evaluate says INSUFFICIENT. v0.1.3 would then pick
> another folder-level sibling; **v0.2 should notice the outgoing link
> and offer `MMLU.md` as the next candidate**.

resolution index, retrieval loop integration, and tests.

## 3. Scope

### Phase 1 — Link resolution index (pure Python, no LLM)

New module `src/pagewiki/wiki_links.py`:

```python
@dataclass(frozen=True)
class ResolvedLink:
    source: TreeNode          # the note that contained [[Alpha]]
    target: TreeNode          # the note Alpha.md resolved to
    raw_target: str           # "Alpha" (as parsed)
    section_anchor: str | None  # "Methods" from [[Alpha#Methods]], else None

class LinkIndex:
    # Built once per scan; reused across many retrieval queries.
    def resolve(self, target: str) -> list[TreeNode]: ...
    def outgoing(self, node_id: str) -> list[ResolvedLink]: ...
    def backlinks(self, node_id: str) -> list[ResolvedLink]: ...
    def dangling(self) -> list[tuple[str, str]]: ...  # (source_id, missing_target)
    def stats(self) -> LinkIndexStats: ...             # for `scan --show-graph`


def build_link_index(
    root: TreeNode,
    *,
    title_matcher: Callable[[str, str], bool] = _default_match,
) -> LinkIndex: ...
```

The index is computed once in `vault.scan_folder` (or as a side-call
after it) and attached to the tree root. Retrieval reads it; scan
commands can print summaries from it.

### Phase 2 — Retrieval loop integration

Two design options, starting with the simpler one:

- **Option A — automatic expansion (recommended for v0.2).** After
  `parse_evaluate_response` returns `INSUFFICIENT`, walk
  `link_index.outgoing(picked.node_id)` and add any still-unvisited
  targets to a `link_candidates` pool. On the next iteration, merge
  that pool into the folder-level candidate list and run
  `select_node_prompt` as usual. The LLM sees both "siblings under the
  current cursor" and "notes linked from recent evidence".
- **Option B — explicit FOLLOW action.** Add a third response shape
  to the evaluate prompt: `FOLLOW: <target_title>`. Requires a new
  parser, a new prompt, and guardrails against invalid target names.
  **Defer to v0.2.1** if option A's ToC review quality turns out to be
  too noisy.

Under option A, the existing parser contract, prompt templates, and
`chat_fn` DI all stay unchanged. The only diff is a few lines inside
`run_retrieval` that read the index and seed `cursor.children`-like
candidate pools.

### Phase 3 — Breadcrumb & cycle prevention

`visited_ids` already prevents re-visiting. The only thing that
changes in v0.2 is how the breadcrumb is displayed. Today:

    Research > paper.md > Methods

After following a link:

    Research > paper.md > Methods → MMLU.md

Proposed: add a `link_chain: list[tuple[str, str]]` field to
`TraceStep` so logs can show both the hierarchical path and the
link-hop history separately. `path_so_far` stays a clean folder
breadcrumb; link hops get their own line in the query log.

### Phase 4 — CLI & logging

- `pagewiki scan --show-graph`: after the 3-tier summary table, print
  a second table with `link_index.stats()`:
  - total outgoing links
  - total dangling links
  - top 10 most-linked-to notes
  - top 10 most-outgoing notes
- `pagewiki ask` query log (`.pagewiki-log/{ts}.md`) gets a new
  `followed_links:` section listing each `[[target]] → note_id` jump
  made during retrieval. Useful for auditing "why did the answer pull
  in notes I didn't expect".

### Phase 5 — Tests

New test files:

- `tests/test_wiki_links.py` (pure-Python, no LLM)
  - Exact title resolution
  - Case-insensitive title resolution (reuses `_normalize_title`)
  - Ambiguous title resolution (see §6 question 1)
  - Dangling link (target missing)
  - Section anchor stripping (`[[Alpha#Methods]]`)
  - Alias stripping (`[[Alpha|별명]]`)
  - `outgoing` / `backlinks` / `stats`
  - Cycle (A → B → A) — index returns both directions; retrieval's
    `visited_ids` is expected to break the cycle.
- `tests/test_retrieval_with_wiki_links.py`
  - Scripted scenario: query → Note A insufficient → Note A has
    outgoing link to Note B → next `select_node_prompt` presents B →
    SUFFICIENT → final answer cites B.
  - Dangling link does not crash the loop.
  - `visited_ids` breaks a 2-cycle in a bounded number of iterations.
  - Query log contains the expected `followed_links:` block.

Target: roughly **12 new test cases**, keeping the suite wall-clock
under 0.4s.

## 4. Non-goals for v0.2

- ❌ **Graph visualization UI.** Obsidian already has one; pagewiki
  stays CLI-first.
- ❌ **Transitive expansion on a single iteration.** Each retrieval
  iteration can follow at most one hop from each already-gathered
  note. Unlimited fan-out would blow up the LLM context and is easy
  to add later if needed.
- ❌ **Tag-based cross-reference.** `#tag` navigation is out of
  scope — v0.2 is wiki-links only. Tags land in a later version.
- ❌ **Renaming a note and auto-updating dangling links.** Detection
  only; no mutation.

## 5. Rollout

Two release shapes to choose from:

- **A. One big v0.2** (Phases 1–5 together). Cleaner story, longer
  single PR, one version bump.
- **B. Staged v0.1.4 → v0.2.** Phase 1 ships as `v0.1.4` (link index
  + `--show-graph`, retrieval untouched). Phases 2–5 ship as `v0.2`
  after Phase 1 is validated against a real vault via
  `scripts/benchmark_vault.py`.

Shape B is safer for a solo project because it lets the real-vault
benchmark catch any resolution-rule surprises before the retrieval
loop starts acting on them.

**Decision (post-session):** **B, staged.** See §6.3.

## 6. Open questions → decisions

All three questions were answered at the end of the v0.1.3
post-merge planning session. The summary table below is the
authoritative answer; the detailed rationale is kept in each
subsection so a future contributor can see the trade-offs.

| # | Question | Decision | Why |
|---|---|---|---|
| Q1 | Title ambiguity | **(2) Return all matches, let the LLM pick** | Simple API, no "distance" logic to design, defers judgment to the model. Obsidian-style nearest-folder ranking stays a v0.2.1 sort-key addition if real-vault benchmarks show the extra candidates hurt quality. |
| Q2 | Section anchor | **(3) Best-match fallback** | Dominant choice: the `_normalize_title` helper added in v0.1.3 makes section matching free, and the note-root fallback keeps the loop safe against typos. Strictly better than either (1) or (2) alone. |
| Q3 | Release shape | **(B) Staged: v0.1.4 (Phase 1) → v0.2 (Phases 2–5)** | Solo-project PR-size sensitivity + Q1's answer depends on real-vault data that `scan --show-graph` can produce *before* retrieval is wired to the index. Same checkpoint-first pattern that v0.1.3 used successfully. |

### Q1 — Title ambiguity: how should `[[Alpha]]` resolve when multiple notes have `title == "Alpha"`?

Obsidian's built-in resolver uses "closest folder relative to the
source note" plus a case-insensitive exact-title match. Options:

1. **Obsidian-compatible**: nearest-folder-first, fall back to any
   match. Pro: matches user expectation from Obsidian. Con: more
   logic, needs "distance" definition for cross-vault or
   sibling-folder ties.
2. **Return all matches, let the LLM pick**: `LinkIndex.resolve`
   returns `list[TreeNode]`, retrieval presents all of them as
   candidates. Pro: simple, pushes judgment to the model. Con: extra
   LLM call per ambiguous link.
3. **First match by sorted node_id**: deterministic, no LLM cost, but
   silently wrong when the "other" Alpha was the intended one.

**Recommendation:** start with (2) (list-returning API) and let the
retrieval loop surface all candidates to the LLM. Add an
Obsidian-compatible ranker as a sort key in v0.2.1 if benchmarks
show the extra candidates hurt quality.

**Decision:** **(2).** `LinkIndex.resolve` returns `list[TreeNode]`;
the retrieval loop adds every element as a candidate for the next
`select_node_prompt` call.

### Q2 — Section anchors: `[[Alpha#Methods]]` should jump where?

v0.1.3 already namespaces section node_ids as `<vault_rel_path>#<zfill_id>`,
but a wiki-link section anchor is the section's human title, not the
zfill id. Options:

1. **Resolve to the note, ignore the anchor.** Simplest. Retrieval
   lands on Alpha.md's root; LLM picks the right section on the next
   descent hop. Adds one extra `select_node_prompt` call.
2. **Resolve to the section.** Walk Alpha.md's Layer 2 tree, find the
   first section whose title (normalized) matches "Methods", return
   that `TreeNode`. Zero extra LLM calls. Fails silently if the
   anchor has a typo.
3. **Best-match fallback.** Try section match first; if it misses,
   fall back to the note root.

**Recommendation:** (3). It's strictly more useful than (1) and has
the same safety profile as (2). The normalization helper already
exists in `pageindex_adapter._normalize_title`.

**Decision:** **(3).** Section match first (via the existing
normalizer), fall back to the note root on miss. The normalizer
should be promoted out of `pageindex_adapter` into a shared
module — probably a new `src/pagewiki/_text.py` — so both the
adapter and `wiki_links.py` can import it without creating a
cross-module cycle.

### Q3 — Release shape: `v0.1.4` intermediate or straight to `v0.2`?

See §5. Recommendation: **B. staged** — `v0.1.4` for Phase 1 only
(purely additive, no retrieval behavior change), then `v0.2` for
Phases 2–5. This gives a real-vault checkpoint between "index
works" and "retrieval uses the index".

**Decision:** **(B).** v0.1.4 scope locked to:

1. `src/pagewiki/_text.py` — extract `_normalize_title` /
   `_titles_match` from `pageindex_adapter` so the new module can
   reuse them without a cycle.
2. `src/pagewiki/wiki_links.py` — `LinkIndex`, `ResolvedLink`,
   `build_link_index`, `resolve` / `outgoing` / `backlinks` /
   `dangling` / `stats`.
3. `cli.py` — `scan --show-graph` flag that prints the
   `LinkIndex.stats()` table after the 3-tier summary.
4. Tests — Phase 5 link-index subset only (~7 cases). No retrieval
   test changes.

Explicitly **out of scope for v0.1.4**: any edit to
`retrieval.py`, `prompts.py`, `logger.py`, or
`tests/test_retrieval*`. Those all land in v0.2 once the index is
validated against the real Obsidian vault via
`scripts/benchmark_vault.py`.
